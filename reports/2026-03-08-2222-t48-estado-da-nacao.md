## 2. Implementacao T48

### 2.1 Alerta automatico de review da allowlist

Implementado no workflow com base em `review.nextReviewAt`:
- novo step: `Build CI summary and allowlist review alert`;
- estados operacionais: `ON_TRACK`, `DUE_SOON` (<=14 dias), `OVERDUE`, `MISSING`;
- alerta sempre visivel em log + GitHub Step Summary;
- nao bloqueia merge por si so (politica mantida).

Arquivos:
- `.github/workflows/assets-audit.yml`
- `tools/assets-ci-summary.js` (novo)

### 2.2 Summary operacional curto no GitHub Actions

Summary passa a exibir:
- status geral do gate;
- quantidade de warnings;
- estado de revisao da allowlist;
- proxima data de review;
- packs afetados pelo diff;
- report de auditoria utilizado.

Arquivo:
- `tools/assets-ci-summary.js` (novo)

### 2.3 Definicao e acao para assets criticos

Politica formalizada para assets criticos de runtime.

Classes criticas adotadas:
- `public/arcade/<slug>/manifest.json`
- `public/arcade/<slug>/bg/**`
- `public/arcade/<slug>/player/**`
- `public/arcade/<slug>/obstacles/**`
- `public/arcade/<slug>/pickups/**`
- `public/arcade/<slug>/ui/**`
- `lib/games/arcade/*assets.ts`
- `lib/games/assets/**`

Classes nao criticas para trigger seletivo:
- `fx/`, `audio/`, docs e templates.

Implementacao no CI:
- novo step de deteccao: `Detect critical asset changes`;
- se houver mudanca critica: executa `npm run test:assets-smoke`;
- sem mudanca critica: smoke adicional e pulado para reduzir ruido.

Arquivos:
- `.github/workflows/assets-audit.yml`
- `tools/assets-critical-smoke-gate.js` (novo)
- `package.json` (`assets:critical-smoke-check`)

## 5. Riscos restantes

- smoke seletivo ainda roda suite completa de assets (nao segmentada por pack), com custo de tempo potencial em PRs maiores;
- classificacao de criticidade ainda pode evoluir por historico real de incidentes;
- metadata da allowlist depende de disciplina de atualizacao periodica (agora com alerta automatico, mas sem bloqueio).

## 6. Recomendacao explicita para T49

1. adicionar opcao de politicas de review (`warn-only` vs `fail-on-overdue`) via variavel de workflow para ambientes mais rigorosos;
2. evoluir smoke seletivo para escopo por pack afetado (mantendo fallback para suite completa quando houver duvida);
3. adicionar metrica simples de tempo de job para validar que reducao de ruido/custo esta funcionando;
4. manter higiene incremental da allowlist sem remocao cega.

## 7. Conclusao

T48 fecha as lacunas operacionais restantes do pipeline de assets sem regressao funcional: alerta de review automatizado, summary de CI mais util e smoke seletivo para mudancas criticas. O hub permanece estavel para `tarifa-zero`, `mutirao` e `cooperativa`, com CI mais previsivel e menos sujeito a dependencias manuais.
